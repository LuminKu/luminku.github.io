类之间的关系是面向对象编程（OOP）中的核心概念，用于描述不同类之间的交互与结构。根据当前主流权威资料（如腾讯云、CSDN、知乎等平台2021–2026年发布的技术文章），类之间的关系通常分为六种：继承（泛化）、实现（Realization）、依赖、关联、聚合、组合。这些关系按耦合强度从弱到强排序为：

> 依赖 < 关联 < 聚合 < 组合 < 继承 ≈ 实现

---

六种关系详解

1. 依赖（Dependency）
- 含义：临时使用，偶然性、弱关系。一个类在某个方法中使用另一个类，但不“拥有”它。
- 代码体现：方法参数、局部变量、静态方法调用。
- UML表示：虚线 + 箭头（指向被依赖类）。
- 示例代码：
  ```java
  class Person {
      void writeCode(Computer c) { // Computer 作为方法参数
          System.out.println("用电脑写代码");
      }
  }
  class Computer {}
  ```

2. 关联（Association）
- 含义：长期、结构性关系，双方“知道”对方存在，通常是平等的“has-a”关系。
- 代码体现：成员变量（引用）。
- UML表示：实线 + 箭头（可单向或双向）。
- 示例代码：
  ```java
  class Person {
      private Shoes shoes; // 成员变量
      public void setShoes(Shoes s) { this.shoes = s; }
  }
  class Shoes {}
  ```

3. 聚合（Aggregation）
- 含义：关联的特例，表示“整体-部分”关系，但部分可独立存在，生命周期不依赖整体。
- 代码体现：成员变量，通常通过构造器或Setter注入。
- UML表示：空心菱形 + 实线（菱形指向整体）。
- 示例：公司与员工、汽车与轮胎。
- 示例代码：
  ```java
  class Engine {}
  class Car {
      private Engine engine;
      public Car(Engine e) { this.engine = e; } // 聚合：Engine可被多个Car共享
  }
  ```

4. 组合（Composition）
- 含义：更强的“整体-部分”关系，部分不能脱离整体存在，生命周期由整体控制。
- 代码体现：成员变量，通常在构造函数中通过 `new` 创建。
- UML表示：实心菱形 + 实线（菱形指向整体）。
- 示例：人与大脑、订单与订单项。
- 示例代码：
  ```java
  class Brain {}
  class Person {
      private Brain brain = new Brain(); // 组合：Brain随Person创建和销毁
  }
  ```

5. 继承（Generalization / Inheritance）
- 含义：“is-a”关系，子类继承父类的属性和方法。
- 代码体现：Java 中用 `extends` 关键字。
- UML表示：空心三角实线（子类 → 父类）。
- 示例代码：
  ```java
  class Animal {}
  class Dog extends Animal {} // Dog 是一种 Animal
  ```

6. 实现（Realization）
- 含义：类实现接口定义的行为契约。
- 代码体现：Java 中用 `implements` 关键字。
- UML表示：空心三角虚线（类 → 接口）。
- 示例代码：
  ```java
  interface Flyable {
      void fly();
  }
  class Bird implements Flyable { // Bird 实现 Flyable 接口
      public void fly() { System.out.println("Bird is flying"); }
  }
  ```

---

关键区别总结

| 关系类型 | 生命周期依赖 | 典型语义 | 代码形式 | 耦合强度 |
|----------|--------------|----------|----------|----------|
| 依赖 | 无 | “uses-a” | 方法参数/局部变量 | 最弱 |
| 关联 | 无 | “has-a”（平等） | 成员变量 | 弱 |
| 聚合 | 无（部分独立） | “has-a”（整体可共享部分） | 成员变量（外部注入） | 中 |
| 组合 | 有（部分随整体消亡） | “contains-a” | 成员变量（内部 `new`） | 强 |
| 继承 | 有（编译绑定） | “is-a” | `extends` | 很强 |
| 实现 | 有（契约绑定） | “is-like-a” | `implements` | 强 |

> 💡 注意：聚合与组合在语法上完全相同（都是成员变量），区别仅在于语义和生命周期设计。关联与聚合的区别也主要在语义（是否强调整体-部分）。

---

实战建议
- 优先使用组合/聚合而非继承，降低耦合，提高灵活性（设计原则：组合 > 继承）。
- 依赖关系最松散，适合工具类、临时调用。
- 接口实现用于定义行为规范，支持多态。
- 在 UML 设计中，正确使用菱形、箭头和虚实线，清晰表达设计意图。
