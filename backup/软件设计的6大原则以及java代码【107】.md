软件设计的六大原则（常称为 SOLID + 迪米特法则 + 开闭原则，但实际六大通常包含开闭）是面向对象编程中指导高质量、可维护、可扩展代码的核心准则。结合当前权威公开资料（截至2026年4月），六大原则及其 Java 代码示例如下：

---

1. 单一职责原则（Single Responsibility Principle, SRP）
> 一个类应该只有一个引起它变化的原因，即只负责一项职责。

Java 示例：
```java
// 职责分离：用户业务、日志、邮件
public class UserService {
    public void registerUser(String name) { /* 注册逻辑 */ }
}

public class LogService {
    public void log(String msg) { /* 日志记录 */ }
}

public class EmailService {
    public void sendWelcomeEmail(String email) { /* 发送邮件 */ }
}
```
违反示例：一个类同时处理用户注册、日志和发邮件，修改任一功能都可能影响其他功能 。

---

2. 开闭原则（Open/Closed Principle, OCP）
> 对扩展开放，对修改关闭。新增功能通过扩展实现，而非修改已有代码。

Java 示例：
```java
interface Shape {
    double calculateArea();
}

class Circle implements Shape {
    private double radius;
    public Circle(double r) { this.radius = r; }
    @Override public double calculateArea() { return Math.PI * radius * radius; }
}

class Rectangle implements Shape {
    private double w, h;
    public Rectangle(double w, double h) { this.w = w; this.h = h; }
    @Override public double calculateArea() { return w * h; }
}

// 新增形状无需修改 AreaCalculator
class AreaCalculator {
    public double totalArea(List<Shape> shapes) {
        return shapes.stream().mapToDouble(Shape::calculateArea).sum();
    }
}
```
新增 `Ellipse` 只需实现 `Shape` 接口，无需改动 `AreaCalculator` 。

---

3. 里氏替换原则（Liskov Substitution Principle, LSP）
> 子类必须能替换其父类而不影响程序正确性。

违反示例：
```java
class Rectangle {
    protected double width, height;
    public void setWidth(double w) { this.width = w; }
    public void setHeight(double h) { this.height = h; }
}

class Square extends Rectangle {
    @Override public void setWidth(double w) {
        super.setWidth(w);
        super.setHeight(w); // 破坏父类行为
    }
}
```
测试中用 `Rectangle rect = new Square(); rect.setWidth(5); rect.setHeight(10);` 期望面积 50，实际得 100，违反 LSP 。

正确做法： 使用组合或接口（如 `Quadrilateral`），避免不合理的继承。

---

4. 依赖倒置原则（Dependency Inversion Principle, DIP）
> 高层模块不应依赖低层模块，二者都应依赖抽象；抽象不应依赖细节，细节应依赖抽象。

Java 示例：
```java
interface UserRepository {
    void save(String user);
}

class JdbcUserRepository implements UserRepository {
    public void save(String user) { System.out.println("Saved to DB: " + user); }
}

class UserController {
    private final UserRepository repo; // 依赖抽象
    public UserController(UserRepository repo) { this.repo = repo; }
    public void register(String name) { repo.save(name); }
}
```
通过构造注入，可轻松替换存储实现（如改为 Redis）。

---

5. 接口隔离原则（Interface Segregation Principle, ISP）
> 客户端不应被迫依赖不需要的方法；应使用多个专门接口，而非一个臃肿接口。

Java 示例：
```java
interface Flyable { void fly(); }
interface Swimmable { void swim(); }

class Duck implements Flyable, Swimmable {
    public void fly() { System.out.println("Duck flies"); }
    public void swim() { System.out.println("Duck swims"); }
}

class Fish implements Swimmable {
    public void swim() { System.out.println("Fish swims"); }
}
```
避免将 `fly()` 强加给不会飞的 `Fish` 。

---

6. 迪米特法则（Law of Demeter / Least Knowledge Principle）
> 一个对象应尽量少地了解其他对象，只与直接朋友通信。

Java 示例：
```java
class Engine {
    void start() { System.out.println("Engine started"); }
}

class Car {
    private Engine engine = new Engine();
    public void start() { engine.start(); } // 封装细节
}

class Driver {
    public void drive(Car car) {
        car.start(); // 正确：只与 Car 交互
        // car.getEngine().start(); // 错误：违反迪米特法则
    }
}
```
避免链式调用（如 `getEngine().start()`），降低耦合 。

---

总结对比表
| 原则 | 核心思想 | 关键词 | 作用 |
|------|----------|--------|------|
| 单一职责 | 一类一事 | 职责聚焦 | 降低复杂性，减少变更风险 |
| 开闭原则 | 扩展不修改 | 抽象 + 扩展 | 适应需求变化，保障稳定性 |
| 里氏替换 | 子可替父 | 继承契约 | 保证继承体系正确性 |
| 依赖倒置 | 依赖抽象 | 面向接口 | 降低模块耦合 |
| 接口隔离 | 小而专 | 精简接口 | 提高灵活性，避免冗余 |
| 迪米特法则 | 少知道 | 降低耦合 | 减少依赖链，提升内聚 |

> 这些原则并非教条，应根据项目规模和变化频率灵活运用。当类职责过多、接口庞大或修改牵一发而动全身时，就是应用这些原则的时机 。
